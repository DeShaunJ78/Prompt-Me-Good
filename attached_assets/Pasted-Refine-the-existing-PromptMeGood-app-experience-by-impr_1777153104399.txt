PART 3 — ADD CLEAR TEXT BUTTONS
========================

Current issue:
Users have to hold backspace to clear long text fields, which is tedious.

Add small clear buttons to the following areas.

1. Your Goal field

Button text:
“Clear”

Behavior:
- Clears only the goal field.
- Does not reset the whole form.
- Does not clear extra details, generated prompt, fine-tune text, or history.

2. Extra Details field

Button text:
“Clear”

Behavior:
- Clears only the extra details field.
- Does not reset the whole form.

3. Generated Prompt output box

Button text:
“Clear Prompt”

Behavior:
- Clears only the current generated prompt/output.
- Resets manual edit state.
- Clears the undo stack for the current prompt.
- Does not erase saved prompt history.
- Does not clear the goal field, extra details field, or fine-tune field unless existing app logic requires it. Prefer not to clear unrelated fields.

4. Fine-Tune Your Prompt field

Button text:
“Clear”

Behavior:
- Clears only the fine-tune instruction field.
- Does not clear the generated prompt.

Styling:
- Use small secondary/ghost-style buttons.
- Keep them subtle.
- Do not let them visually compete with “Generate Prompt” or “Copy Prompt.”
- Keep tap targets comfortable on mobile.
- Place each button near the related field label or inside the field header area if that is cleaner.

Implementation notes:
- Only show a clear button when the related field has content, if that fits the existing UI style.
- If always visible, keep disabled or subtle when the field is empty.
- Add accessible labels if needed, such as `aria-label="Clear goal"`.

Do not:
- Accidentally clear saved history.
- Reset unrelated fields.
- Break controlled input state.
- Break manual edit or undo behavior.

========================
PART 4 — PROMPT HISTORY CONSOLIDATION
========================

Current issue:
Prompt History becomes very large when users generate many prompts because too much information and too many actions are visible at once.

Improve Prompt History so it is compact by default while preserving all existing features.

1. Compact default history rows

Prompt History should show a compact list by default.

Each saved prompt row/card should show:
- Name/title if available, otherwise the first line of the prompt.
- Date/time.
- Small preview snippet.
- Favorite indicator if applicable.
- Tags if they already exist and can fit cleanly.

Keep each row visually compact.

2. Hide actions by default

Move the existing prompt actions into an expanded row or “More Actions” area.

Actions that must remain available:
- Restore
- Copy
- Favorite
- Save As Template
- Duplicate
- Add Name
- Edit Tags
- Add Note
- Archive
- Delete
- Remix

These actions should not all be fully visible by default.

Use one of these patterns:
- “More Actions” button that reveals the actions for that prompt.
- Expand/collapse row that reveals the actions and details.
- Existing accordion/disclosure component if the app already uses one.

Do not remove any actions.

3. Add “Clear All History” button

Add a button near the Prompt History controls.

Button text:
“Clear All History”

Behavior:
- Requires confirmation before deleting.
- Clears all saved prompt history from localStorage or from the existing history persistence mechanism.
- Updates any counts/stats if the app displays them.
- Shows toast:
“Prompt history cleared”

Confirmation dialog text:
“Clear all saved prompts? This cannot be undone.”

Confirmation buttons:
- “Cancel”
- “Clear All”

Important:
- Do not clear generated prompt output unless existing app behavior intentionally ties it to history. Prefer clearing history only.
- Do not clear templates unless templates are literally stored as part of the same prompt history array. If templates are separate, leave them alone.
- Do not clear favorites separately unless favorites are part of the saved prompt history data.

4. Add optional “Collapse All”

If multiple prompt rows can be expanded at once, add a “Collapse All” control near the history controls.

Behavior:
- Collapses all expanded history rows.
- Does not delete or modify history.

If the implementation only allows one row expanded at a time, “Collapse All” is optional and can be skipped.
